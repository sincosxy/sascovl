# CargoFlow — Личный кабинет заказчика (MVP)

## Суть проекта
Система для пошагового оформления заявок на перевозку грузов (Wizard form). 
Регистрация только через администратора.

## Структура проекта (Плоская)
- `app/main.py` — Точка входа FastAPI и роуты.
- `app/models.py` — Все таблицы БД (User, CargoOrder) + SQLAlchemy Base.
- `app/schemas.py` — Pydantic модели (валидация данных).
- `app/db.py` — Настройка асинхронного движка и сессий БД.
- `app/config.py` — Чтение .env через Pydantic Settings.
- `app/auth.py` — Логика безопасности (хеширование, JWT).
- `alembic/` — Миграции базы данных.

## Стек
- **Backend:** Python 3.10+, FastAPI, SQLAlchemy (Async), Alembic.
- **Database:** PostgreSQL (в Docker) + pgAdmin (порт 5050).
- **Environment:** WSL2, venv, .env (игнорируется гитом).

## Запуск инфраструктуры
1. База: `docker-compose up -d`
2. Миграции: `alembic upgrade head`
3. Создание админа: `python create_first_admin.py` (скрипт в корне)


# CargoFlow — Backend (Current State)

## Что реализовано:
1. **Инфраструктура:** 
   - PostgreSQL + pgAdmin в Docker.
   - Миграции через Alembic (настроены на асинхронный драйвер для проекта и синхронный для миграций).
2. **База данных:**
   - Таблицы `users` (с ролями user/admin) и `cargo_orders`.
3. **Безопасность:**
   - Хеширование паролей через `bcrypt` (прямое использование библиотеки).
   - JWT-авторизация (создание и валидация токенов).
   - Интеграция со Swagger (замочек авторизации через OAuth2PasswordRequestForm).
4. **Инструментарий:**
   - Скрипт `create_first_admin.py` для инициализации системы.

## Как запустить после клонирования:
1. Поднять базу: `docker-compose up -d`
2. Создать venv и поставить зависимости: `pip install -r requirements.txt`
3. Применить миграции: `alembic upgrade head`
4. Запустить сервер: `python -m uvicorn app.main:app --reload`

## Текущие эндпоинты:
- `POST /api/login` — получение токена (принимает Form Data).
- `POST /api/orders` — создание пустой заявки-черновика (требует авторизации).
- `GET /api/me` — проверка текущего пользователя.

## Статус проекта
Настроена сложная реляционная схема базы данных, поддерживающая два типа перевозок (контейнерные и генгруз) с системой справочников.

## Структура БД (Реализовано)
- **Users & Auth:** Роли (Admin/User), хеширование bcrypt, JWT.
- **Cargo Orders (Header):** Основная информация о поручении. Вместо строк теперь используются связи (FK) на справочники.
- **Справочники (System):**
    - `Ports`: Глобальный список портов.
    - `Equipments`: Типы контейнеров (20DC, 40HC и т.д.).
- **Справочники (User-specific):**
    - `Counterparties`: Грузоотправители/получатели. У каждого пользователя свой список.
- **Детализация (Items):**
    - `Containers`: Для контейнерных перевозок (привязка к Equipment).
    - `GeneralCargoItems`: Для генеральных грузов (габариты, объем).

## Технические особенности
- **Alembic + Postgres Enum:** Применен хак для ручного создания типа `transporttype` в миграциях.
- **Relationships:** Настроены каскадные удаления (удаляешь заказ — удаляются его контейнеры).

## Roadmap на следующий шаг:
1. Написание Seed-скрипта для первичного заполнения `Ports` и `Equipments`.
2. Создание API для "умного" поиска и сохранения контрагентов (Counterparties).
3. Реализация Step-by-Step API для заполнения поручения.

📝 Что реализовано (Status Update)
1. Архитектура пошагового оформления (The Wizard)
Реализован SPA-флоу на базе HTMX: переход между шагами происходит без перезагрузки страницы.
Шаг 1: Выбор типа перевозки (Контейнер/Генгруз) с визуальной подсветкой выбранной карточки.
Шаг 2: Маршрут (порты из глобального справочника) и «умный» поиск контрагентов (Shipper/Consignee/Notify) с автозаполнением.
Шаг 3: Динамическая таблица контейнеров. Реализовано добавление/удаление строк на лету, поддержка SOC (собственных контейнеров), номеров пломб и весов.
2. Умные справочники (Counterparties)
Реализован механизм Get-or-Create: система сама создает нового контрагента в базе пользователя, если его там еще нет, прямо во время сохранения заказа.
Настроен живой поиск (Autocomplete) через datalist, который возвращает HTML-фрагменты с сервера.
3. Управление списком заказов (Dashboard)
Информативная таблица: Вместо ID теперь отображаются иконки (📦/🏗️), названия портов и агрегированная информация по грузу (например, 2 × 40HC).
Интерактивность: Добавлен аккордеон: при клике на стрелочку подгружаются детальные данные о контейнерах без перехода на другую страницу.
Удаление: Реализовано удаление заказов с подтверждением и каскадной очисткой связанных данных в БД.
4. Редактирование и навигация
Память форм: Все эндпоинты поддерживают GET и PATCH. При возврате на любой шаг или открытии старого заказа из списка, все поля (включая селекты и чекбоксы) автоматически заполняются данными из БД.
Breadcrumbs: Навигационная цепочка шагов позволяет быстро перемещаться по этапам оформления.
🛠 Технический стек (текущий)
Backend: FastAPI + SQLAlchemy (Async) + PostgreSQL.
Frontend: HTMX + Tailwind CSS + Jinja2.
State: Хранение черновиков в БД с привязкой к current_user.
🚀 Roadmap на будущее:
Генерация документов: Создание PDF-заявки (Booking Note) на основе введенных данных.
Генгруз: Реализация Шага 3 для негабаритных грузов (вес/объем/места).
Статусы: Перевод заказа из «Черновика» в «Оформлено» с запретом дальнейшего редактирования.

1. Рефакторинг архитектуры Wizard (The "Fork")
Глобальное ветвление: Логика оформления теперь разделена на раннем этапе. На Шаге 1.5 введен выбор собственности оборудования (SOC/COC), который определяет поведение всех последующих шагов.

2. Модернизация логики контейнеров
Наследование свойств: Признак is_soc и инструкции по возврату порожнего перенесены на уровень CargoOrder. Контейнеры теперь наследуют эти свойства от родительского заказа, что упрощает UI.
Динамический UI (Step 3):
Для COC: Скрыты поля ввода номеров контейнеров и чекбоксы собственности. Клиент указывает только типы оборудования и веса.
Для SOC: Поля номеров контейнеров становятся доступными и обязательными для заполнения.
Интерфейс оператора: Реализована возможность аннулирования отдельных контейнеров в заказе с указанием причины. Использован "бесшовный" метод HTMX (перерисовка всей таблицы через hx-select), что позволило сохранить монолитность шаблонов.
3. Технические решения (HTMX & Backend)
State Persistence: Реализовано сохранение и восстановление состояния (radio/checkbox) при навигации «Назад/Вперед» и при редактировании черновиков (проверки на is none в Jinja2).
CSS-оптимизация: Внедрен класс .cancelled-row для управления стилистикой отмененных позиций (зачеркивание, прозрачность) без дублирования условий в каждой ячейке таблицы.
Fixes: Исправлены ошибки UnboundLocalError и TypeError при ручном вызове роутов из других функций FastAPI.
