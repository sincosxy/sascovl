⚓️ Проект CargoFlow: Контекст для ИИ (v. 2.0)
1. Стек и Архитектура:
Backend: FastAPI + SQLAlchemy (Async) + PostgreSQL.
Frontend: HTMX + Tailwind CSS + Jinja2 (монолитные шаблоны, без лишних partials).
Логика: SPA-флоу через подмену блоков в #main-content.
2. Ключевая логика (обновлено 30.04):
Глобальный SOC/COC: Признак собственности выбирается на Шаге 1.5. Он сохраняется в CargoOrder и наследуется всеми контейнерами в заказе.
Динамический Step 3:
Если заказ COC, поля container_number скрыты/readonly (номера дает линия/оператор).
Если заказ SOC, поля номеров открыты и обязательны.
Операторский интерфейс: Реализована аннуляция контейнера (is_cancelled) с указанием причины (cancel_reason). Отрисовка через класс .cancelled-row в manage_order.html.
Ветвление: На Шаге 1.5 генгруз (general_cargo) пролетает мимо контейнерных настроек сразу к маршруту.
3. Структура БД:
CargoOrder: Хранит is_soc, return_instructions, transport_type, статусы и контрагентов.
Container: Хранит equipment_id, веса, пломбы, is_cancelled.
4. Твое правило:
Не делать жестких рамок (валидация веса — только предупреждение, а не запрет).
Минимум лишних файлов верстки.
Клиенту — простота, Оператору — вся мощь правок (разбивка весов, изменение типов и т.д.).
